# Java Actions

Use this when creating or modifying Java actions. Java actions extend Mendix with custom Java logic — use them for complex algorithms, external library integration, file processing, or operations that exceed what microflows can do.

## File Location

Java action implementation files live at:
```
javasource/<modulename_lowercase>/actions/<ActionName>.java
```

Example: `javasource/mymodule/actions/JA_CalculateTax.java`

Business logic goes in a separate implementation layer:
```
javasource/<modulename_lowercase>/impl/<ActionName>Impl.java
```

## Naming Convention

Prefix Java actions with `JA_`:
- `JA_CalculateTax`
- `JA_SendEmail`
- `JA_ProcessFile`

## Writing Java Action Code

Read the existing file first, then write only between the markers:

```java
// BEGIN USER CODE
// Your implementation here — this block is preserved by Studio Pro
return someValue;
// END USER CODE
```

Everything outside those markers is regenerated by Studio Pro and will be overwritten.

## Mendix Type → Java Type Mapping

| Mendix Type | Java Type |
|-------------|-----------|
| String | `java.lang.String` |
| Integer | `java.lang.Long` |
| Decimal | `java.math.BigDecimal` |
| Boolean | `java.lang.Boolean` |
| DateTime | `java.util.Date` |
| Object (entity) | `IMendixObject` |
| List of objects | `java.util.List<IMendixObject>` |

## Core API Patterns

```java
// Create a new object
IMendixObject order = Core.instantiate(context, "MyModule.Order");

// Read an attribute
String name = obj.getValue(context, "Name");

// Write an attribute
obj.setValue(context, "Status", "Processing");

// Commit an object
Core.commit(context, obj);

// Delete an object
Core.delete(context, obj);

// Retrieve objects by XPath
List<IMendixObject> results = Core.retrieveXPathQuery(
    context,
    "//MyModule.Order[Status = 'Pending']"
);

// Log a message
Core.getLogger("MyModule").info("Processing order: " + orderId);
```

## Recommended Structure: Separate Implementation Classes

Keep Java actions as thin wrappers. Put the real logic in `impl/`:

```
javasource/mymodule/
  actions/JA_CalculateTax.java      ← thin wrapper, generated skeleton
  impl/TaxCalculator.java           ← pure Java logic, no Mendix imports
  impl/MendixOrderAdapter.java      ← converts IMendixObject ↔ plain Java
```

This lets you unit test the implementation without the Mendix runtime.

## Error Handling

```java
// BEGIN USER CODE
try {
    // your logic here
    return result;
} catch (Exception e) {
    Core.getLogger("MyModule").error("Failed: " + e.getMessage(), e);
    throw new UserException("Operation failed. See application log.");
}
// END USER CODE
```

## Writing Java Files via Kiro Tools

The MCP server's `write_file` only covers the `/jsactions/` domain. Write Java files using Kiro's own file tools (`Write`, `Edit`, `Read`). Always read the file first before editing.

**Read:**
```
Read javasource/mymodule/actions/JA_MyAction.java
```

**Edit (preserve BEGIN/END markers):**
Use Edit to replace only the content between `BEGIN USER CODE` and `END USER CODE`.

## Checklist

- [ ] Only modify code between `BEGIN USER CODE` and `END USER CODE`
- [ ] Return type matches the declared return type in the action definition
- [ ] Always call `Core.commit()` when saving changed objects
- [ ] Use `Core.getLogger()` instead of `System.out.println`
- [ ] Keep complex logic in `impl/` classes for testability
- [ ] Handle exceptions explicitly — don't let them bubble as raw Java exceptions
